# Ecosystem — shared concepts

Every sibling protocol in the OrangeCheck family (Attest, Lock, Vote, Stamp,
Agent, Pledge) inherits the same plumbing. Learning it once saves re-learning it
six times. This section is the shared layer.

## What's in here

- **[The canonical message](/ecosystem/canonical-message)** — every sibling
  protocol produces a signed "canonical message" in the same shape (header
  line + field lines + ack line, LF line endings, lexicographically sorted
  extensions). Byte-identical across implementations, which is how we can ship a
  TS SDK and a Python SDK that produce the same attestation IDs.
- **[BIP-322 signing](/ecosystem/bip322)** — every protocol signs the canonical
  message via BIP-322. Why not legacy `signmessage`. What wallets support it.
  The one-line answer to "which signing scheme do I use?" is always "BIP-322,
  unless you're on a pre-SegWit P2PKH address."
- **[Nostr kind-30078](/ecosystem/nostr-kind-30078)** — NIP-78
  parameterized-replaceable addressable events are how every sibling publishes
  its artifacts (attestations, device records, stamps, ballots, agent
  delegations, pledges + outcomes). The `d` tag convention is per-protocol; the
  event shape is shared.
- **[Conformance vectors](/ecosystem/conformance)** — every sibling ships
  normative test vectors in its `oc-*-protocol` repo. Reference impls are
  required to pass them on every CI run. This is how we catch implementation
  drift before it reaches npm.
- **[Security model](/ecosystem/security)** — the shared threat model across
  every sibling. What Bitcoin stake proofs DO protect against (sybil at-scale,
  bot spam, anonymous cheap identity farming). What they DON'T (proof of
  personhood, privacy-preserving claims, chain-analysis resistance). Read this
  before shipping anything.

## Why a shared section

Writing the canonical-message format six times (once per protocol) would produce
six slightly-different descriptions and, over time, actual drift in the impls.
By specifying it once and linking from every protocol's pages, we keep the docs
in sync with the reality — every sibling literally uses the same
canonical-message builder under the hood.

When a protocol _extends_ a shared concept — like Vote's secret-mode commitment,
or Lock's envelope canonicalization rule — the protocol's own pages cover only
the delta, and link back here for the baseline.

## Reading order

If you're new:

1. Skim [canonical message](/ecosystem/canonical-message) — 5 minutes.
2. Read [BIP-322](/ecosystem/bip322) if you're not already comfortable with it —
   5 minutes.
3. Skim [security](/ecosystem/security) before shipping anything — 10 minutes,
   high-value.
4. Dive into the protocol you picked in the
   [decision tree](/getting-started/which-protocol).

Kind-30078 and conformance vectors are _reference_ sections — no need to read
them linearly. Come back when an SDK surprises you.
