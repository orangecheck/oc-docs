# OrangeCheck Connect

**Add OrangeCheck to your existing login.** Keep Google, Auth0, Clerk,
email/password, magic links — whatever you already run. After your user
authenticates, your backend makes **one HTTPS call** and that user gets a
Bitcoin-backed OrangeCheck identity (`did:oc`).

There is no new SDK to learn. A Connect integration is a single server-to-server
`POST`.

## Two ways to integrate

| Pathway                      | You do                                                                              | Who owns the login UX |
| ---------------------------- | ----------------------------------------------------------------------------------- | --------------------- |
| **OC-as-login**              | Mount [`@orangecheck/auth-client`](/sdk/auth-client) — OrangeCheck _is_ your login. | OrangeCheck           |
| **OC Connect** _(this page)_ | Keep your own login; call `/api/connect/provision` after it.                        | You                   |

## How it works

1. Register a Connect app at
   [me.ochk.io/me/developer/connect](https://me.ochk.io/me/developer/connect) —
   you get a `client_id`.
2. Your user signs in however they already do.
3. Your backend calls `POST https://ochk.io/api/connect/provision` with proof of
   who the user is.
4. OrangeCheck returns a `did:oc`. Store it against your user record.

The provisioned account is **federation-custodied** — exactly like an email-OTP
signup. The user can later graduate to self-custody. OrangeCheck never holds a
Bitcoin key it cannot account for.

## Two trust tiers

Connect accepts a user identity by one of two means. Configure either or both on
your Connect app.

### Tier 1 — verified OIDC token

You forward a real OIDC ID token (Google today; Apple / Microsoft / GitHub
next). OrangeCheck verifies it itself against the provider's JWKS — signature,
issuer, expiry, **audience binding**, and `email_verified`. OrangeCheck trusts
the _provider_, so the email is genuinely verified and the user resolves to
their **canonical `did:oc`** — the same identity they have everywhere in the
OrangeCheck family.

Register your own Google OAuth client id(s) on the Connect app so OrangeCheck
can bind the token's `aud`.

```ts
// Your backend, after the user signs in with Google.
const res = await fetch('https://ochk.io/api/connect/provision', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
        client_id: process.env.OC_CONNECT_CLIENT_ID,
        tier: 'oidc',
        provider: 'google',
        id_token: googleIdToken, // the ID token from your Google sign-in
    }),
});
const result = await res.json();
// → { ok: true, did_oc: 'did:oc:…', assurance: 'L1', is_new, upgrade_available: false }
```

### Tier 2 — signed assertion

For any login flow that is _not_ a verifiable OIDC provider — email/password,
magic links, your own SSO — you sign a small claims object with an Ed25519 key.
OrangeCheck trusts _you_ for the fact of authentication.

Generate the key in the Connect console: it is created in your browser, you save
the private half, and only the public half reaches OrangeCheck.

Sign the **exact bytes** of the JSON you send — there is nothing to
canonicalize:

```ts
import { createPrivateKey, sign } from 'node:crypto';

// Load the base64 private key you saved from the console. A raw 32-byte
// Ed25519 key becomes a Node key object with this PKCS#8 prefix.
const PKCS8_PREFIX = Buffer.from('302e020100300506032b657004220420', 'hex');
const privateKey = createPrivateKey({
    key: Buffer.concat([
        PKCS8_PREFIX,
        Buffer.from(process.env.OC_CONNECT_KEY, 'base64'),
    ]),
    format: 'der',
    type: 'pkcs8',
});

const claims = {
    external_user_id: user.id, // your own stable user id
    email: user.email, // optional — treated as UNVERIFIED
    issued_at: Math.floor(Date.now() / 1000),
    nonce: crypto.randomUUID(), // single-use
};
const claimsBytes = Buffer.from(JSON.stringify(claims), 'utf8');
const signature = sign(null, claimsBytes, privateKey);

const res = await fetch('https://ochk.io/api/connect/provision', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
        client_id: process.env.OC_CONNECT_CLIENT_ID,
        tier: 'assertion',
        claims_b64: claimsBytes.toString('base64url'),
        signature: signature.toString('base64url'),
    }),
});
const result = await res.json();
// → { ok: true, did_oc: 'did:oc:…', assurance: 'L0', is_new, upgrade_available: true }
```

## What you get back

```json
{
    "ok": true,
    "did_oc": "did:oc:…",
    "assurance": "L1",
    "is_new": true,
    "upgrade_available": false
}
```

Store `did_oc` against your user record — that is the whole integration. You now
know "my user ↔ this OrangeCheck identity" and can show it, let them graduate to
self-custody, or call other OrangeCheck APIs on their behalf.

## Assurance — and why Tier 2 is L0

Every result carries an `assurance` level:

| Level  | Meaning                                                                                                              |
| ------ | -------------------------------------------------------------------------------------------------------------------- |
| **L1** | A verified identifier — Tier 1 (OrangeCheck checked the provider itself). Resolves to the user's canonical `did:oc`. |
| **L0** | Integrator-asserted — Tier 2. OrangeCheck verified the _fact_ you authenticated them, but nothing about the email.   |

A Tier-2 (`L0`) result is an **integrator-scoped** identity: it is keyed to your
app and **cannot** touch a higher-assurance account. This is deliberate — it
means a Connect integration can never assert its way into someone's existing
OrangeCheck identity. An `L0` user reaches their canonical `did:oc` later, by
verifying a shared identity (an email one-time code, or a Bitcoin signature);
`upgrade_available: true` flags that.

## Notes

- `/api/connect/provision` is **server-to-server** — call it from your backend,
  never the browser. The ID token / assertion is the proof.
- Both tiers are **single-use**: a replayed token or assertion is rejected. Use
  a fresh `nonce` per assertion.
- Manage your apps — audiences, keys, suspend — any time at
  [me.ochk.io/me/developer/connect](https://me.ochk.io/me/developer/connect).
